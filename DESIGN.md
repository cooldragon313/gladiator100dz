| 檔案 | 功能 |
|------|------|
| `stats.js` | 玩家數值、派生屬性計算、UI 渲染 |

### 現有戰鬥流程（當前）
- **純點按式回合制**
- 玩家點 [攻擊] / [特技] / [防禦] → 敵人 AI 自動行動 → 等待下一次點擊
- `Battle.doAction(action)` 為主入口
- 自動模式：`_autoTimer` setInterval 驅動

### 現有技能條系統（已完成）
```
三條路線，由武器的 route 欄位決定：
  rage  (怒氣)：被打 +15、被暴擊 +25、命中 +10
  focus (集中)：命中 +12、暴擊 +22、閃躲 +20；被打 -10
  fury  (盾牆)：被命中 +10、格擋 +28、防禦姿態 +25

特技（gauge = 100 時釋放）：
  rage  → 血怒：ATK×2, DEF×0.5, EVA=0，持續 3 回合
  focus → 絕對瞬息：必中, CRT+40, SPD×1.5，持續 2 回合
  fury  → 盾牆反擊：下次被打完全格擋 + 反擊×1.5，1 回合（即時反應型）
```

### 現有武器路線對應
```
dagger     → focus
hammer     → rage
shortSword → fury
spear      → focus
longSword  → fury
warHammer  → rage
heavyAxe   → rage
```

## ATB 戰鬥時間系統

### 設計確認
- 採用 **FFVI ATB 條**風格
- 每個單位有獨立的 ATB 條（0–100），與技能條分離
- ATB 條滿 → 該單位行動 → ATB 條歸 0 重跑
- 全局 tick 計數器（最大 120 ticks = 兩小時時限）

### 揮速（swingTime）定義
> swingTime 越小 = 揮速越快 = ATB 條填充越快

| 武器 | swingTime |
|------|:---------:|
| 匕首 | 2 |
| 短劍 | 3 |
| 長槍 | 4 |
| 槌   | 5 |
| 長劍 | 5 |
| 長槌 | 8 |
| 重斧 | 9 |

### ATB 填充公式
```js
// 每 tick 填充量
atbFillRate = unit.derived.SPD / weapon.swingTime

// SPD 由 TB_calcDerived 算出（AGI、DEX、武器加成等）
```

### Tick 迴圈結構（待實作）
```
setInterval(tick, 50ms)：  // 50ms per tick
  1. 所有存活單位 atbBar += atbFillRate
  2. 若敵人 atbBar >= 100：
       敵人行動（AI），atbBar 歸 0
  3. 若玩家 atbBar >= 100：
       暫停迴圈（clearInterval）
       顯示行動選項
  4. globalTick++
  5. 更新 UI 條

玩家選擇行動後：
  執行行動
  玩家 atbBar 歸 0
  重啟迴圈（setInterval）
```

### 需加入 TB_WEAPONS 的新欄位
```js
// 每個武器物件加入：
swingTime: 2,  // 數字，參考上方表格
```

## 評分系統

### 適用範圍
- **競技場戰鬥**：使用下方評分表，戰後顯示評級並計算加成
- **劇情強制戰**：不計評分，依戰鬥過程影響後續對話內容（事件評語）

### 評分表（競技場）

| 等級 | 條件 | 名聲乘數 | 其他加成 |
|------|------|:--------:|---------|
| S 迅雷 | ≤60 ticks 結束，HP殘餘 >70% | ×2 | 主人好感+高、長官好感+高 |
| A 穩健 | ≤100 ticks 結束，HP殘餘 >40% | ×1.5 | — |
| B 消耗 | ≤120 ticks 結束 | ×1 | — |
| C 殘喘 | 超時 OR HP殘餘 <20% | ×0.7 | 可能扣體力 |

### 計算時機
```
戰鬥結束（敵人 HP=0）→ 計算評分 → 顯示評級 → 進入砍首選擇面板
砍首選擇結果 → 覆蓋/追加到評分加成上
最終結算名聲與好感
```

## 技能條與大招

### 雙門檻系統（新設計）
> 舊系統只有一個 100% 門檻。新系統加入 50% 小技能 + 100% 大招。

#### rage 路線（怒氣蓄積型，被打越打越強）
| 門檻 | 技能名 | 效果 | delay |
|:---:|--------|------|:-----:|
| 50% | 爆筋 | 下次攻擊 ATK+60%，立即 | 0 |
| 100% | 血怒 | ATK×2, DEF×0.5，持續 3 ticks | 0 |

#### focus 路線（精準蓄積型，命中越多越強）
| 門檻 | 技能名 | 效果 | delay |
|:---:|--------|------|:-----:|
| 50% | 瞬刺 | 保證命中 + CRT+20%，立即 | 1 tick |
| 100% | 絕對瞬息 | 必中 + CRT+40 + SPD×1.5，持續 2 ticks | 0 |

#### fury 路線（防禦蓄積型，格擋越多越強）
| 門檻 | 技能名 | 效果 | delay |
|:---:|--------|------|:-----:|
| 50% | 強架 | 下次格擋強制成功（一次性） | 0 |
| 100% | 盾牆反擊 | 被打時完全格擋 + 反擊×1.5（即時反應） | 0（reactive） |

### Delay 機制
```
delay 0：立即執行，無風險
delay 1：ATB 條再跑 1 個 swingTime 才判定
          期間被打可打斷（不消耗技能條）
delay 3：重型武器大招（例：重斧「決斷斬」）
          蓄力期間 DEF 下降，成功則傷害爆炸

weapons 各自的大招 delay 建議：
  匕首（三連刺）  → delay 1
  槌（震盪錘）    → delay 2
  短劍（貫穿）    → delay 1
  長槍（突刺）    → delay 1
  長劍（強斬）    → delay 2
  長槌（山崩）    → delay 3
  重斧（決斷斬）  → delay 3
```

### 技能觸發規則
```
- 技能可在任何時間點觸發（不受 ATB 條限制）
- 但技能有 delay：delay > 0 時進入「蓄力中」狀態
- 蓄力中：ATB 繼續跑，敵人可能搶先行動打斷
- 打斷後：技能取消，技能條不消耗（保留）
```

## 主手/副手雙槽系統

> 2026-04-13 設計討論確認。取代舊的「盾牌待修問題」段落。

### 設計動機

舊系統的路線（rage/focus/fury）完全由武器的 `route` 欄位決定，但 fury 路線的技能名為「強架」「盾牆反擊」——名字帶"盾"卻與是否裝盾無關。這導致：

- 匕首＋鐵盾 → focus，拿著盾卻在「瞬刺」？
- 槌＋鐵盾 → rage，拿盾卻「血怒」狂砍？
- 短劍不帶盾 → fury，沒盾卻「盾牆反擊」？

**結論：路線應由「裝備組合」決定，而非單一武器欄位。**

### 裝備槽位

```
主手（equippedWeapon）：武器（單手 or 雙手）
副手（equippedOffhand）：盾牌 / 單手武器 / 空
  - 雙手武器佔兩手 → 副手強制為 null
  - 單手武器 + 盾牌 = 盾衛流
  - 單手武器 + 單手武器 = 雙持流
  - 單手武器 + 空手 = 武器原始路線
```

### 路線判定規則

| 主手 | 副手 | 最終路線 | 理由 |
|---|---|---|---|
| 任何單手武器 | **盾牌** | **fury** | 有盾 → 盾牆流，強架/盾牆反擊名符其實 |
| 單手武器 | **單手武器（雙持）** | **rage** | 雙武器 → 瘋狂進攻，爆筋/血怒合理 |
| 單手武器 | 空 | 武器原始 route | 匕首=focus, 槌=rage, 短劍=fury |
| 雙手武器 | （不可裝備） | 武器原始 route | 長槍=focus, 長劍=fury, 長槌/重斧=rage |
| 空手 | 空 | rage | 拳頭=蠻力 |

### 雙持（Dual Wield）代價與加成

| 項目 | 效果 | 理由 |
|------|------|------|
| ATK | 副手武器 ATK × 50% 加入總攻擊 | 副手不如主手有力 |
| SPD | −3 | 兩把武器協調困難 |
| BLK | 強制歸 0 | 雙手握武器無法格擋 |
| BpWr | 強制歸 0 | 同上 |
| ACC | −5 | 注意力分散 |
| 路線 | 鎖定 **rage** | 雙持=純攻擊，爆筋/血怒 |

### 帶盾加成

| 項目 | 效果 |
|------|------|
| BLK / DEF | 盾牌原有的 BLK + DEF（不變） |
| 路線 | 鎖定 **fury** |
| 50% 技能 | 強架（受擊 −80%）— 盾擋 |
| 100% 技能 | 盾牆反擊 — 完美呼應 |

### 三種玩法定位

| 打法 | 裝備搭配 | 路線 | 風格 |
|---|---|---|---|
| **盾衛** | 單手＋盾 | fury | 防守反擊，穩健持久 |
| **狂戰** | 雙持 / 重鈍器 | rage | 暴力輸出，不顧防禦 |
| **刺客** | 匕首空手 / 長槍 | focus | 精準暴擊，蓄力爆發 |

### 盾牌數據（TB_SHIELDS，不變）

| id | 名稱 | BLK | DEF | SPD |
|----|------|----:|----:|----:|
| `none` | 無 | 0 | 0 | 0 |
| `woodShield` | 木盾 | 5 | 2 | 0 |
| `ironShield` | 鐵盾 | 9 | 4 | -2 |

### 實作要點

1. `Stats.player.equippedShield` → `equippedOffhand`（可存盾牌 ID 或單手武器 ID）
2. `TB_calcDerived()` 根據 `offhandId` 判斷盾/武器/空 → 覆蓋路線
3. 雙持時副手 ATK×0.5 進衍生值，BLK/BpWr 歸 0
4. `Stats.calcDerived()` 同步鏡像邏輯
5. 雙手武器裝備時自動清空副手；裝備盾牌/副手武器時若主手為雙手則提示
6. 存檔版本升級，向下相容舊 `equippedShield` 欄位

## 遊戲願景與核心定位

### 類型定位
**Narrative Roguelite（敘事型 Roguelite）**
- 框架固定（100 天 / 競技場），每次填入的內容不同
- 參考：Hades（多局推進敘事）、Banner Saga（選擇不可逆）、Undertale（路線分歧）

### 核心原則

1. **硬核 = 選擇有後果，後果不可逆**
   - 砍了隊友 → 永遠死了，任務線斷了
   - 暗殺失敗 → 世界記住了，不是讀檔重來
   - 選了某背景 → 某些結局天然容易，某些幾乎不可能
2. **100 天是框架，不是限制**
   - 不是「撐過 100 天就贏」
   - 而是「100 天內你能活出什麼樣的故事」
3. **一次遊玩看不到全部內容**
   - 不同背景、不同選擇、不同 NPC 生死 → 不同事件觸發
   - 多周目才能拼出完整世界觀
4. **不殺人也能活，但更難**
   - 和平路線：名聲成長慢、領主不喜歡，但隊友好感飆升
   - 參考 Undertale：和平路線比暴力路線需要更多技巧與耐心
5. **自由度高，但有代價**
   - 玩家可以走主線（贏得最終戰役）
   - 也可以逃亡、暗殺、買回自由、或發現隱藏真相
   - 每條路線有自然的吸引力，不需要強迫

### NPC 工廠函式（寫一次不改）
```js
function buildEliteNPC(id, currentDay) {
  const def = ELITE_DEFS[id];
  const growth = ARCHETYPE_GROWTH[def.archetype];
  const daysPassed = Math.max(0, currentDay - def.arriveDay);
  const stats = {};
  for (const attr of ['STR','DEX','CON','AGI','WIL','LUK']) {
    stats[attr] = def.baseStats[attr]
      + Math.floor(daysPassed * (growth[attr] || 0))
      + randomInt(-1, 1);
  }
  return { ...def, id, stats, hp: calcHpFromCON(stats.CON),
           alive: true, affection: 10, storyFlags: [] };
}
```

## 結局架構

### 主結局（4 條）

| 結局 | 觸發條件 | 基調 |
|---|---|---|
| **冠軍之路** | Day 100 存活 + 最終戰勝利 + 名聲 ≥ 80 | 榮耀。贏得自由，但永遠是制度的一部分 |
| **復仇** | 發現真相 + 擊殺/推翻領主 | 痛快但代價巨大（可能失去盟友） |
| **逃亡** | 3+ 隊友好感 ≥ 70 + 逃亡道具齊全 + 名聲 ≤ 40 | 自由但卑微。活了，但世界不記得你 |
| **買回自由** | 金錢/交易條件達成（政治手腕或特殊背景） | 和平但妥協 |

### 隱藏結局（2 條）

| 結局 | 觸發條件 | 基調 |
|---|---|---|
| **起義** | 殺死領主 + 5+ 隊友存活 + 名聲 ≥ 60 | 斯巴達克斯式，帶所有人走 |
| **真相** | 流浪漢背景 + 所有身世線索收集 | 發現自己是誰，但答案比不知道更痛苦 |

### 失敗結局

| 結局 | 觸發條件 |
|---|---|
| **死亡** | 戰鬥中 HP 歸 0 且無復活條件 / Day 100 最終戰敗 |

### 和平路線設計
- 不殺人 → 名聲成長慢、領主不滿，但隊友好感飆升
- 饒恕的人可能成為盟友（逃亡路線關鍵人力）
- 被饒恕者危機時回報（特殊事件）
- 「踩臉」作為中間選項（不殺但維持名聲）

```

|---|---|---|---|

| **鐵獄** | 中階 | ★★★ | 正規訓練、裝備一般、競爭適中，「變強或死去」 |
| **血冠殿** | 頂級 | ★★★★★ | 裝備精良、全是殺人機器、淘汰殘酷，「弱者撐不過第一週」 |

### 訓練所影響
```
改變的：
  ① 初始場地配置（不同 FIELDS）

```

- 每 10 層 Boss（菁英隊友強化版、已故 NPC 幻影等）
- 每 5 層選臨時增益（Roguelike 式）
- 死亡結算分數

```

  全屬性 -3、訓練效率 -30%、戰鬥 SPD -5

  ① 自然恢復（等 4 天）
  ② 醫生（新 NPC「瑟拉斯」）→ 好感越高恢復越快

## 框架先行五大原則

> **做對這五點，未來加任何系統都不用重構。**

### 原則 1：Flag 管理器要最早做且可序列化
```js
// 統一接口，所有故事狀態透過它存取
Flags.set('truth_discovered', true);
Flags.has('marcus_told_story');
Flags.getAll();              // 序列化用
Flags.loadFrom(savedFlags);  // 反序列化用

// 禁止：任何系統直接操作 Stats.player 上的旗標欄位
// 強制：所有故事狀態集中在 Flags 模組
```

### 原則 2：NPC 結構一次預留所有欄位
```js
// 即使現在不用，欄位先存在 → 未來加系統不需要遷移
const npc = {
  id, name, title, desc,
  archetype, personality, profession, level,
  hiddenTalent, religion, faction,
  baseAffection, scars: [], dreams: [],
  portrait: null,             // 🎨 美工預留
  voiceLines: {},
  questlineId: null,
  alive: true, storyFlags: [],
};
```

### 原則 3：事件系統資料驅動，不用函式驅動
```js
// ✅ 對：事件是純資料物件
const STORY_EVENT = {
  id, conditions: {...}, onTrigger: {...}
};

// ❌ 錯：事件是 function（無法從資料文件加載）
function marcusEvent() { if (...) {...} }
```

### 原則 4：存檔 schema 預留完整
```js
// 不要每加一個系統就 v5/v6/v7
// 預留所有未來欄位，版本只處理大改動
const saveSchema = {
  version: 5,
  player: {...},
  flags: {},              // 預留
  quests: {},             // 預留
  worldState: {},         // 預留
  religion: {},           // 預留
  scars: [],              // 預留
  dreams: [],             // 預留
  faction: null,          // 預留
  rumors: [],             // 預留
};
```

## 新系統補充說明

> **以下為討論過但尚未寫入 Part B 的系統的簡介。詳細設計在實作前補充。**

### S4 世界狀態系統
6 種世界狀態，開局隨機決定，影響整局基調：
- **peaceful 和平年代**：默認，無特殊 modifier
- **warTime 戰亂之時**：強盜事件 ×4、競技場名聲 ×1.3、裝備漲價、可能觸發 C7 戰爭徵召
- **plague 瘟疫之年**：生病率 ×2、訓練所封鎖、NPC 可能病死
- **famine 飢荒之年**：食物消耗加快、廚娘食物效果 ×0.5、黑市活躍
- **festivalYear 盛世慶典**：名聲 ×1.5、最終戰獎勵 ×2、觀眾更嗜血（難度 -1）
- **politicalCoup 王權動盪**：領主好感波動、規則改變事件、暗殺更容易

### C7 戰爭徵召事件（5~10 天特殊事件）
觸發：世界狀態 = warTime && Day 30+ && 名聲 ≥ 40
- **階段 1 出發**（1 天）：編入步兵隊
- **階段 2 行軍**（2~3 天）：行軍格取代訓練格，老兵聊天、照顧傷員、埋伏戰
- **階段 3 戰役**（1 天）：3v3 或連續小戰
- **階段 4 返回**（1 天）：帶回戰利品/新隊友/傷勢/PTSD
好友 NPC 可能一起參戰，永久死亡風險。複用現有戰鬥系統，不需新戰略層。

### P5 預製角色（Kenshi 式）
通關特定結局後解鎖固定身份的開局角色：
- 固定名字、立繪、屬性、裝備、背景故事
- 覆蓋背景/訓練所/季節選擇
- 有專屬事件和結局加成
- 例：**斷劍賽恩**（前騎士，臉疤-20 好感，但 STR 16 起手）
- 例：**蜜拉治癒者**（強制和平路線，50 次饒恕解鎖）

### S6 多維度命運抽取
開局 UI 顯示多個維度的難度星數，而非單一總分：
- 生存難度（天氣+世界狀態）
- 社交難度（背景特質）
- 戰鬥難度（初始屬性+對手強度）
- 資源難度（職業 NPC 缺失）
- 綜合難度：⭐~⭐⭐⭐⭐⭐
新手保護：第一次遊玩限制難度 ≤ 6，通關後漸進解鎖。

### E8 宗教信仰系統
6 個神祇 + 無信仰選項：
- **瑪爾斯**（戰神）：戰鬥 buff，解鎖「殉道者」結局
- **佛圖娜**（命運女神）：LUK+ 加成，身世揭露加成
- **涅墨西斯**（復仇女神）：對砍過你的敵人 +30% 傷害，限復仇背景
- **席瓦努斯**（森林之神）：AGI+、逃脫加成，解鎖「回歸荒野」結局
- **維斯塔**（爐火女神）：治療隊友、家園之道，解鎖「守爐人」結局
- **無信仰**：WIL+3，不限制結局
機制：每日祈禱行動、信仰任務、敵對教徒好感衝突、信仰結局。

### C1 疤痕與永久傷害
戰鬥重傷留下永久疤痕：
- 臉疤（威嚇+、好感-）、跛腳（AGI-）、獨眼（ACC-/CRT+）
- 失指（武器受限）、斷肋（CON-）、內傷（體力上限-）、PTSD（心理崩潰）
- 普通醫生治普通傷、大師級治重傷、心理傷靠信仰/好友/時間

### E7 派系系統
訓練所內部政治（範例：鐵獄）：
- **老兵派**（傳統、紀律）/ **新人派**（互助、反抗）/ **獨行派** / **權力派**
- 加入 → 該派好感大加成，對立派 -
- 某些任務需要派系推薦
- 結局判定：逃亡需新人派、奪權需權力派

### C2 夢境系統
每晚 10% 觸發：
- 記憶夢（amnesia 背景）→ 揭露過去
- 預言夢 → 暗示接下來的事件
- 噩夢 → 砍人多會被亡魂糾纏
- 神諭夢 → 宗教任務
- 思念夢 → 家人/好友

### C4 婚姻/伴侶系統
候選包含 dagiSlave、melaKook、外來商人、某菁英、醫生瑟拉斯等：
- 階段：陌生→認識→信任→好感→喜歡→愛慕→承諾
- 解鎖結局：雙人逃亡、贖身同行、犧牲、對方為你死
- 關鍵：奴隸環境中必須有代價和阻礙

### C3 謠言系統
你的行為有機率成為謠言，在 NPC 間擴散：
- 正面謠言：隊友好感+
- 負面謠言：隊友-、長官+
- 可購買/散佈/阻止謠言
- 暗殺目擊者會傳話 → 警戒提升

### C5 酒/藥物 Vice 系統
三種物質：短期強化、長期代價

function TB_calcDamage(atk, def) {
  const part = rollBodyPart(atk.weapon.hitBias);
  const partArmor = getPartArmor(def, part);
  const partDEF = partArmor ? partArmor.eqBonus.DEF : 0;
  
  let damage = atk.derived.ATK - partDEF;
  if (part === 'head') damage *= 1.5;       // 頭部加成
  if (!partArmor) damage *= 1.5;             // 無防護爆傷
  
  // 重傷 → 疤痕機率
  if (damage > def.hp * 0.3 && def.hp <= def._hpMax * 0.4) {
    rollScar(def, part, damage);
  }
  
  return { damage, part };
}

## D.6 經驗值/技能點/特性 Tier 系統

> 採用實況野球模式：訓練 → EXP → 屬性升級 → SP → 購買特性。
> 特性分 5 個 Tier，需要多屬性門檻。

### 屬性成長曲線

```js
// 每個屬性獨立 EXP
Stats.player.exp = {
  STR: 0, DEX: 0, CON: 0, AGI: 0, WIL: 0, LUK: 0,
};

// 升級成本：指數成長
function expToNext(currentLevel) {
  return Math.ceil(10 * Math.pow(1.15, currentLevel - 10));
}
// STR 10→11: 10 EXP
// STR 15→16: 20 EXP
// STR 20→21: 40 EXP
// STR 25→26: 81 EXP
// STR 30→31: 163 EXP
```

### 訓練行動改寫

```js

stdTraining:    { expMultiplier: 1.3 },

```

### EXP 計算公式

```js
finalExp = baseExp 
         * expMultiplier       // 場地
         * moodMult             // 心情 0.75~1.25
         * traitMult            // 特性加成
         * (1 - penaltyPct)     // 體力懲罰
         * rng(0.9, 1.1);       // 隨機浮動
```

## 🆕 D.12 NPC 故事揭露系統（Story Reveals）

> **狀態**：2026-04-15 設計定案，第一階段（卡西烏斯範本）實作中。
> 把 NPC 故事從「好感度數字」轉變為「由玩家角色特性觸發的漸進敘事」。
> 核心哲學：**不預告未來，但給方向感；每個玩家看到的故事都不一樣**。

### 核心概念

1. **故事 = 碎片陣列**：每個 NPC 有 `storyReveals[]`，包含多段 reveal（flavor + event）
2. **漸進揭露**：好感門檻逐步解鎖，高好感才看到更深的東西
3. **特性共鳴**：某些 reveal 只對特定玩家開放——失眠症、神經質、識字等特性作為鑰匙
4. **內容可漸進補**：NPC 沒填 storyReveals 就用 fallback，允許一次只補一個 NPC
5. **故事長度自由**：陣列可短可長，每個 NPC 的深度不需要一致
6. **道具是故事載體**：重要 reveal 可贈送獨特道具（如符牌），物品身上帶 storyTag 可被其他 NPC 的事件檢查到，形成**故事網**（見 D.14）

### 資料結構

```js
cassius.storyReveals = [
  // ── Flavor 段：關係圖卡片常駐顯示 ─────────
  {
    id:        'cassius_quiet',
    type:      'flavor',
    affection: 10,
    text:      '他很少說話。',
  },

  // ── Event 段：夜間隨機觸發（一次性） ─────
  {
    id:              'cassius_whisper_night',
    type:            'event',
    affection:       40,
    requireAnyTrait: ['insomnia_disorder', 'neurotic'],
    chance:          0.3,
    onceOnly:        true,
    text:            '深夜裡你聽見他低聲念著一個名字...',
    logColor:        '#8899aa',
  },

  // ── Event 段：觸發時贈送道具 ─────
  {
    id:              'cassius_charm_touch',
    type:            'event',
    affection:       80,
    requireMinAttr:  { WIL: 15 },
    chance:          0.2,
    onceOnly:        true,
    text:            '他把那塊符牌塞進你手裡...',
    grantItem:       'marcoCharm',
  },
];
```

### 支援的觸發條件

| 欄位 | 意義 |
|------|------|
| `affection` | 最低好感門檻（必填） |
| `requireAnyTrait` | 玩家需擁有其中任一特性 |
| `requireAnyAilment` | 玩家需擁有其中任一病痛 |
| `requireMinAttr` | 屬性最低值 `{ STR: 15 }` |
| `requireFlag` | 需特定故事旗標 |
| `requireOrigin` | 只對某個背景開放 |
| `requireItemTag` | 玩家護符/物品上的 storyTag |
| `chance` | 事件型的觸發機率（0~1） |
| `onceOnly` | 觸發過寫入 `player.seenReveals[]` 永不再觸發 |

### Fallback

NPC 沒填 `storyReveals` 或沒有符合條件的 reveal → 關係圖卡片顯示「你對他知之甚少」。

## 🆕 D.15 跨系統整合檢查清單 (Integration Checklist)

> **為什麼這個章節存在**：2026-04-15 開發過程中發現，每次加新功能常常隔一段時間才意識到「喔，這應該也連到好感/物品/故事/旗標...」。
> 這不是設計疏失，而是 emergent design 的本質——遊戲意義是湧現的，不是預先列舉的。
>
> **這個清單的用途**：實作任何新功能（事件、NPC、動作、物品、特性）之前，跑一遍這 8 問，把該有的橫向連結在**設計階段**就想到，而不是事後補。
>
> **誰負責**：AI 在提案階段主動自問一遍，寫進建議中。使用者可隨時說「檢查整合清單」強制重跑。
>
> **為什麼不讓 AI 每次讀完整 DESIGN.md**：文件 6000+ 行，讀全文浪費 context 且不精準。精準的辦法是 grep 相關段落 + 跑這份穩定的橫向清單。

### 📋 整合檢查清單（實作前必問 8 題）

當你要加一個新的**事件、動作、NPC、物品、特性、選項**時，依序問：

1. **好感度連動** — 這個功能會不會改變某個 NPC 的好感？如果應該改，是 NPC 主動（他喜歡你這樣做）還是被動（你沒做會扣）？有沒有好感門檻才能觸發？
2. **特性共鳴** — 有沒有哪個玩家特性應該讓這個功能**觸發不同結果**？（識字解鎖讀書、神經質強化夜間事件、失眠症共鳴等）
3. **故事揭露連動** — 這個功能會不會解鎖某個 NPC 的 story reveal？或者反過來——某個 story reveal 應該導致這個功能出現新選項？
4. **資源/物品流動** — 這個功能會不會給/花 EXP、金錢、護符、個人物品、SP？給的東西有沒有 `storyTag` 能觸發其他 NPC 的反應？
5. **Stage 演出需求** — 這個功能的敘事**有沒有足夠重量**需要 Stage 小過場（`Stage.playEvent`）？還是只寫 log 就夠？重要事件要加 `addLog(..., true)` 的 flash。
6. **Origin 差異化** — 這個功能可以按玩家背景不同而不同嗎？農家子弟看到跟貴族看到是否應該不同？至少要考慮過，不做也要主動說「這個不分 origin」。
7. **觸發頻率** — 這功能是 `onceOnly`、每日、每週、機率觸發、還是永久常駐？如果是機率型，有沒有條件能提升機率（共鳴加成）？
8. **旗標與後續鉤子** — 這個功能需要設一個 `Flags.set(...)` 讓**未來系統**能檢查「玩家做過 X」嗎？常見需要旗標的情境：故事分歧、一次性選擇、跨天記憶。

### 使用範例：errand_market 事件

問這 8 題會得到：

| 題目 | 答 | 行動 |
|------|---|------|
| 1. 好感度連動 | **是**——主人派你辦事本身代表信任 | errand 加 `masterArtus +3`；門檻 `master ≥ 20` 才派 |
| 2. 特性共鳴 | 神經質 → 市集看到更多細節；識字 → 看得懂告示牌 | 特性為選項後續擴充方向 |
| 3. 故事揭露 | 完成 errand 可能觸發主人的 story reveal | 主人有 storyReveals 後補上 |
| 4. 資源流動 | 給 food/mood/money（已做） | 已有 |
| 5. Stage 演出 | **是**——這是罕見的「離開訓練場」事件 | 用 `queueStageEvent + playEvent` 小過場（已做）|
| 6. Origin 差異化 | 貴族看到老朋友、農家子弟看到舊村莊風景 | 待擴充 |
| 7. 觸發頻率 | 目前每 7 天一次 | 可改為好感門檻 + 機率 |
| 8. 旗標 | `heard_arena_rumor`（已設） | 已有 |

**結論**：現在的 errand 漏了 1 和 6。其他都有做或已規劃。

### 這個流程不是官僚主義

8 題跑完 30 秒內。AI 在提案時用這清單做「一次心算橫向掃描」，不用真的寫進每次回覆。  
**出現問題時就更新清單**——如果哪一次發現某個新維度不在這 8 題裡，就加第 9 題。  
清單本身會成長，但緩慢、穩定、可控。

### 協力倍率公式

在場隊友中，好感度達門檻者會加入協力計算：

```js
let mult = 1;
partners_in_field.forEach(partner => {
  const aff = teammates.getAffection(partner.id);
  if      (aff >= 60) mult *= 1.6;  // 夥伴
  else if (aff >= 30) mult *= 1.3;  // 認識
  // < 30：不參與協力
});
// mult 即最終倍率，套用在 EXP 獲得量上
```

### 倍率對照表

| 在場人數 | 全認識(30+) | 全夥伴(60+) |
|:--------:|:-----------:|:-----------:|
| 0 | 1× | 1× |
| 1 | 1.3× | 1.6× |
| 2 | 1.7× | 2.6× |
| 3 | 2.2× | 4.1× |
| 4 | 2.9× | 6.6× |
| 5 | 3.7× | **10.5×** |

### 視覺分級

| 倍率 | 動作特效 | 日誌文字顏色/大小 |
|------|----------|-----------------|
| 1× | 無 | 白色小字 |
| < 2× | 1–2人綠光 | 綠色 |
| 2×–4× | 藍光 + 閃爍 | 藍色放大 `⚡ 協力爆擊！` |
| 4×–10× | 大藍 + 畫面震動 | 藍色大字 `💥 超協力爆擊！` |
| 10×+ | 爆框 + 全畫面閃 | 紫色超大字 `🌟 傳說爆擊！` |

### 協力台詞觸發

- 認識(30+)：出現普通協力台詞（1–2句）
- 夥伴(60+)：出現熱情台詞 + 泡泡放大發光
- 3人協力達藍光：每人都說一句，泡泡依序出現（cascade 動畫）

### 影響的訓練項目

協力倍率套用在 **EXP 增益**上，不影響基礎屬性 delta：
```
baseExp = action.effects[].delta × expMultiplier（裝備/特性加成）
finalExp = baseExp × synergyMult
```

## D.11.1 統一 assets 欄位規範

所有內容模板都必須包含 `assets` 物件，預留美術與音效槽位。**即使現在用不到，欄位也要存在**，避免未來擴展時遷移所有資料。

### 標準 assets 結構

```js
assets: {
  portrait:   null,   // 立繪/頭像（PNG/WebP 帶透明）
  icon:       null,   // 小圖示（UI 列表/狀態欄）
  cg:         null,   // 全螢幕 CG（事件/結局的強調畫面）
  background: null,   // 場景/卡片背景圖
  bgm:        null,   // 背景音樂（loop）
  sfx: {              // 音效組（物件，可延伸任意鍵）
    activate:   null,   // 觸發/使用
    ambient:    null,   // 環境/持續
    notify:     null,   // 通知/解鎖
    // 可因內容類型添加其他鍵：greet/death/happy/alarm/swing/hit/equip 等
  },
},
```

### 欄位意義

| 欄位 | 用途 | 未填時 fallback |
|---|---|---|
| `portrait` | 大圖（NPC 立繪、寵物圖、宗教神像） | 文字卡片 + 剪影圖 |
| `icon` | 小圖（32~64px，列表/狀態用） | 第一個中文字元 |
| `cg` | 全螢幕（重大事件/結局/開場） | 文字敘述全螢幕 |
| `background` | 場景背景（訓練所/天氣色調） | CSS gradient |
| `bgm` | 背景音樂循環 | 靜音或保持前一首 |
| `sfx.*` | 短音效 | 靜音 |

### 檔案 fallback 規則

```js
// UI 讀取規則範例
function renderPortrait(npc) {
  if (npc.assets?.portrait) {
    return `<img src="${npc.assets.portrait}" alt="${npc.name}">`;
  }
  // fallback：顯示首字 + 灰底
  return `<div class="portrait-fallback">${npc.name[0]}</div>`;
}
```

**原則：缺資產不會讓遊戲崩潰，只會顯示文字替代。**

## D.11.16 NPC 百科顯示規範（漸進揭露）

> 這一段定義 `[所有人]` tab 的顯示規則。
> 五階段漸進 + 負向反向揭露。

### 列表頁

```
世界上有 42 人 · 你已認識 18 人

篩選：[全部] [隊友] [權威] [職業] [其他]
排序：[好感降] [好感升] [最近互動] [姓名]

┌────┐ ┌────┐ ┌────┐ ┌────┐
│立繪│ │立繪│ │立繪│ │ ?? │
│名字│ │名字│ │名字│ │??? │
│❤85 │ │❤62 │ │💔-20│ │    │
└────┘ └────┘ └────┘ └────┘
```

每個卡片顯示：
- 剪影/立繪（依 reveal level）
- 名字或 `???`
- 好感等級 icon（❤ / 💔 / ??）
- 好感數字（可隱藏為只顯示階段）

### 詳細頁（正向揭露對照表）

| Reveal Level | 好感 | 顯示欄位 |
|---|---|---|
| 0 未見過 | — | `???` + 剪影 + 「未知之人」+ 被提及次數 |
| 1 見過 | 0~19 | name, title, desc, role, basic portrait |
| 2 認識 | 20~39 | + baseStats, derived, hp |
| 3 熟悉 | 40~59 | + personalityDesc, traits, faction |
| 4 親近 | 60~79 | + equipment, pets, schedule, hiddenQuestHints['60'] |
| 5 知心 | 80~100 | + background, secrets, hiddenQuestHints['80'] |

### 詳細頁（負向揭露對照表）

| Reveal Level | 仇恨 | 顯示欄位（不同於正向） |
|---|---|---|
| -1 不悅 | −19~−1 | 同「見過」+ 冷淡描述 |
| -2 厭惡 | −39~−20 | + dislikeReason |
| -3 憎恨 | −59~−40 | + **weaknesses** + **fears** |
| -4 仇敵 | −79~−60 | + **schedule vulnerabilities**（刺殺用） |
| -5 不共戴天 | −80~−100 | + **deepest secrets**（復仇線索） |

### 顯示規則

```js
function renderNpcDetail(npcId) {
  const level = NpcCodex.getRevealLevel(npcId);
  const npc   = ELITE_DEFS[npcId] || NPC_DEFS[npcId];

  if (level === 0) return renderUnknown(npc);

  const sections = [];

  if (level >= 1 || level <= -1) sections.push(renderBasic(npc));
  if (level >= 2)                 sections.push(renderAttributes(npc));
  if (level >= 3)                 sections.push(renderTraits(npc));
  if (level >= 4)                 sections.push(renderEquipment(npc));
  if (level >= 5)                 sections.push(renderBackground(npc));

  // 負向揭露
  if (level <= -2) sections.push(renderDislikeReason(npc));
  if (level <= -3) sections.push(renderWeaknesses(npc));
  if (level <= -4) sections.push(renderSchedule(npc, 'vulnerabilities'));
  if (level <= -5) sections.push(renderDeepestSecrets(npc));

  return sections.join('');
}
```

### 顯示原則

- **不顯示精確數字**：「好感度 23」顯示為「認識」
- **不預告未來**：不說「再+10 就能看到特性」
- **給方向感**：可以說「他似乎還有話沒說完」
- **未見過可被提及**：如果其他 NPC 對話提到某人，記錄在 `_mentionedBy[]`，百科顯示「你聽過這個名字 N 次」

## D.11.19 常見錯誤與檢查表

### ❌ 常見錯誤 1：忘記 assets 欄位

```js
// ❌ 錯
marcus: { name: '馬庫斯', ... }

// ✅ 對
marcus: { 
  name: '馬庫斯', 
  ...,
  assets: { portrait: null, icon: null, sfx: {} },
}
```

### ❌ 常見錯誤 2：中文路徑 or 絕對路徑

```js
// ❌ 錯
assets: { portrait: '資產/NPC/馬庫斯.png' }
assets: { portrait: '/home/user/project/asset/...' }

// ✅ 對
assets: { portrait: 'asset/image/npc/marcus.webp' }
```

### ❌ 常見錯誤 3：Flag 命名不一致

```js
// ❌ 錯（大小寫、空白、特殊字元）
Flags.set('Marcus Told Story')
Flags.set('marcusTold')

// ✅ 對
Flags.set('marcus_told_story')
```

### ❌ 常見錯誤 4：特性 Tier 與需求數量不匹配

```js
// ❌ 錯（Tier 3 只需要 1 個屬性 → 太好拿）
perfect_form: {
  tier: 3,
  requirements: { STR: 18 },  // 只有 1 個
  ...
}

// ✅ 對
perfect_form: {
  tier: 3,
  requirements: { STR: 18, DEX: 18, AGI: 15 },  // 3 個
  ...
}
```

### ❌ 常見錯誤 5：任務 step 忘記設 flag

```js
// ❌ 錯
steps: [
  { id: 'step1', event: 'xxx' },  // 沒有 setFlag，下一步無法觸發
  { id: 'step2', trigger: { flag: 'xxx_done' }, ... },
]

// ✅ 對
steps: [
  { id: 'step1', event: 'xxx', setFlag: 'marcus_step1_done' },
  { id: 'step2', trigger: { flag: 'marcus_step1_done' }, ... },
]
```

### ❌ 常見錯誤 6：裝備 hitBias 加總不等於 1.0

```js
// ❌ 錯
hitBias: { head: 0.2, body: 0.5, arms: 0.2, legs: 0.2 }  // = 1.1

// ✅ 對
hitBias: { head: 0.2, body: 0.5, arms: 0.2, legs: 0.1 }  // = 1.0
```

# Part E — 奴隸循環核心設計（Phase 1）⭐

> 2026-04-13 大幅重構討論的結論。這是 Phase 1 的**正式核心設計**，
> 在 Phase 0 基礎設施完成後立即執行。
>
> 此章節定義了遊戲的**主要玩法循環**。
> 會徹底改變左欄、時段系統、事件觸發邏輯、經濟流向。

---

## E.1 核心原則 — 奴隸沒有自主權

### 設計宣言

> **玩家的角色是一個奴隸。**
> **奴隸不能自由選擇要做什麼、和誰見面、吃什麼、什麼時候休息。**
> **玩家的選擇權存在於「如何面對發生的事」，而不是「主動去做什麼事」。**

### 對照表：有 vs 沒有

| 玩家能做 | 玩家不能做 |
|---|---|
| 選擇今天訓練什麼屬性 | 決定何時吃飯 |
| 選擇如何回應事件 | 決定去哪個場地 |
| 選擇事件中的對話態度 | 主動找 NPC 聊天 |
| 選擇是否順從命令 | 主動申請勞動任務 |
| 選擇事件中的消費目標 | 自己去市集花錢 |
| 選擇如何使用時間（訓練 vs 擺爛） | 自己決定拜訪誰 |

### 遊戲節奏的對比

**傳統 RPG**：
```
玩家主動  → [去廚房] → [和梅拉聊天] → 好感 +3
              ↑ 玩家掌控一切
```

**本遊戲**：
```
系統決定  → 早餐時間到 → 自動去廚房
系統提供  → 梅拉看到你 → 事件：「今天多給你半塊肉」
玩家選擇  → [感謝] or [沉默接受] or [推開] → 不同後果
```

### 不是「削弱玩家」，而是「讓選擇變重」

每個選擇變少 → 每個選擇變重。
玩家一天只做 4 次主動決定（訓練選擇），加上 1~3 次事件中的選擇。
總計約 5~7 次決定/天，但每一次都有明確後果。

對照：傳統 RPG 一天可能做 30+ 次決定（點 30 個按鈕），但多數是無意義的「刷」。

## E.2 左欄極簡化 + 時段重解讀

### 新的左欄

```
┌─────────────────────┐
│  訓練菜單              │
├─────────────────────┤
│ 🗡 基礎揮砍 ─ STR    │
│ 🏃 步法練習 ─ AGI    │
│ 💪 耐力訓練 ─ CON    │
│ 🎯 精細動作 ─ DEX    │
│ 🧘 冥想調息 ─ WIL    │
│ 🌙 沉思反省 ─ WIL    │
├─────────────────────┤
│ ◆ 就寢・結束今天      │
└─────────────────────┘
```

**只有訓練 6 項 + 就寢。** 其他全部移除：
- 移除所有動態生成的聊天
- 移除「拜訪長官」、「拜見主人」
- 移除「切磋對練」
- 移除「幫廚娘打雜」、「偷食物」、「觀摩打鐵」、「委託修繕」
- 移除「買食物」、「閒逛市集」等主動動作
- 移除場地切換按鈕

**訓練 6 項對應六維屬性**（現有大部分訓練動作可複用）。

### 時段系統重解讀

原本 8 個時段都是「訓練格」，改為分類：

```
時段     時間           類型         可操作
───────────────────────────────────────────
slot 0   06:00~08:00    早餐事件     自動
slot 1   08:00~10:00    訓練格 1     玩家選
slot 2   10:00~12:00    訓練格 2     玩家選
slot 3   12:00~14:00    午餐事件     自動
slot 4   14:00~16:00    訓練格 3     玩家選
slot 5   16:00~18:00    訓練格 4     玩家選
slot 6   18:00~20:00    晚餐事件     自動
slot 7   20:00~22:00    休息/事件    混合
───────────────────────────────────────────
         22:00~        就寢事件     自動
```

**每天玩家主動選擇 4 次訓練。** 其他時段由系統控制。

### 場地系統的變更

| 行為 | 現況 | 新設計 |
|---|---|---|
| 場地切換 | 玩家點按鈕 | ❌ 移除（UI 隱藏） |
| 玩家位置 | 自由選 | 由事件/預設決定 |
| 場景背景 | 依場地切換 | 依當前事件 / 預設 |
| 訓練場預設 | 玩家要走過去 | ✅ 預設場景 |

**預設狀態**：玩家在「訓練場」。
**事件觸發**：事件把玩家「帶到」目標場地（廚房/鐵匠鋪/主人房）。
**事件結束**：自動回到訓練場。

玩家「看得見」其他場地（場景可以顯示），但無法主動進入。

## E.3 用餐與休息自動化

### 三餐自動觸發

```
早餐 07:00 → 自動事件 meal_breakfast
午餐 12:00 → 自動事件 meal_lunch
晚餐 18:00 → 自動事件 meal_dinner
```

**預設行為（安靜自動）**：
- 顯示一行 log：`「你排隊領了一份粥。food +20, mood +2」`
- 不打斷遊戲流程
- 不彈事件視窗

**只有異常狀態才彈事件**（見 E.4）。

### 就寢自動觸發

```
22:00 → 自動事件 sleep_night
```

**預設行為**：
- 顯示 log：`「你倒在床上。一夜無夢。」`
- 自動結算隔日（stamina 回滿、food 微降、mood 微升）
- 推進天數

**異常狀態彈事件**（體力太滿 → 失眠；心情太低 → 惡夢）。

### 廚娘的故事載體

因為每日 3 餐 × 100 天 = **300 次用餐機會**，廚娘（梅拉）成為全遊戲接觸最多的 NPC。

```
梅拉的故事透過「用餐事件變體」漸進展開：

好感  0~10  → 冷漠、機械分餐
好感 10~30  → 偶爾看你一眼、多舀半勺
好感 30~50  → 輕聲提醒、多給一塊肉
好感 50~70  → 藏私房菜給你、開始說點什麼
好感 70~90  → 講她女兒、講她過去
好感 90+    → 觸發主線任務：梅拉的秘密
```

每個階段的用餐事件從事件池隨機抽取（同階段內有多個變體），
所以即使同一階段也不會重複乏味。

**梅拉的任務線不需要「主動拜訪」，用餐就是她的舞台。**

## E.4 數值閾值與事件觸發表

### 核心原則：**正常靜默，異常彈事件**

玩家數值在正常範圍時，完全無事件，安靜推進。
只有數值偏離正常範圍時才觸發對應事件。

### 閾值定義

```js
const THRESHOLDS = {
  food: {
    overStuffed:  85,    // ≥85 → 跳餐事件（太飽）
    full:         60,    // 60~84 → 正常
    normal:       35,    // 35~59 → 正常
    hungry:       15,    // 15~34 → 訓練效率 -20%
    starving:      5,    // 5~14 → 強制偷竊事件
    critical:      0,    // ≤4  → 崩潰倒地
  },
  stamina: {
    overcharged:  90,    // ≥90 → 失眠事件（精力太旺）
    high:         60,    // 60~89 → 正常
    normal:       30,    // 30~59 → 正常
    tired:        15,    // 15~29 → 擺爛事件可觸發
    exhausted:     5,    // 5~14 → 強制休息事件
    collapse:      0,    // ≤4  → 訓練中暈倒
  },
  mood: {
    euphoric:     80,    // ≥80 → 訓練爆發事件（+效率）
    happy:        50,    // 50~79 → 正常
    normal:       25,    // 25~49 → 訓練效率 -15%
    sad:          10,    // 10~24 → 自暴自棄事件
    depressed:     0,    // ≤9  → 強制放棄訓練
  },
};
```

### 事件機率表

```js
const EVENT_TRIGGER_RATES = {
  // 跳餐（飽食過高時）
  skip_meal:        { condition: 'food >= 85', chance: 0.35 },

  // 飢餓偷竊（低飽食）
  hunger_steal:     { condition: 'food <= 14',  chance: 1.0 },  // 強制
  hunger_minor:     { condition: 'food <= 30',  chance: 0.25 }, // 輕度

  // 失眠（體力過滿）
  insomnia:         { condition: 'stamina >= 90 && hour >= 22', chance: 0.30 },

  // 疲憊擺爛
  training_slacker: { condition: 'stamina <= 25 && action == training', chance: 0.25 },
  fatigue_force:    { condition: 'stamina <= 10', chance: 1.0 },

  // 心情低落
  mood_slack:       { condition: 'mood <= 20 && action == training',  chance: 0.30 },
  mood_despair:     { condition: 'mood <= 5',  chance: 1.0 },

  // 心情爆發
  mood_flow:        { condition: 'mood >= 80 && action == training',  chance: 0.40 },
};
```

**所有數字都是可調參數**，從一個 constants 檔案讀取，隨時能平衡。

### 事件範例：跳餐

```js
event: skip_meal_temptation
  trigger: 用餐時段 + food >= 85 + 隨機 35%
  narrative: "鐘聲響了。你摸了摸肚子，其實並不餓。"
  choices:
    [ 乖乖去排隊 ]   → food +20, mela +1
    [ 跳過去訓練 ]   → food -5, 多一次訓練機會, flag 'skipped_meal_voluntarily'
    [ 偷偷溜走 ]     → 隱形擲骰, 成功時得訓練機會 + 'slacker' 標籤
```

### 事件範例：飢餓偷竊

```js
event: hunger_steal_critical
  trigger: food <= 14 + 下一餐前
  forced: true
  narrative: "你的肚子痛得像在燃燒。下一餐還有很久。"
  choices:
    [ 忍著 ]                → stamina -15, mood -15, HP -5
    [ 偷硬饅頭 ]            → 85% 成功, food +10
    [ 偷熱湯 ]              → 55% 成功, food +25, 被抓 mela -10
    [ 偷主人的晚餐雞 ]      → 20% 成功, food +40, 被抓嚴重懲罰
    [ 敲梅拉的門求情 ]      → 好感 ≥ 40 時成功, 否則好感 -5
```

### 事件範例：失眠之夜

```js
event: insomnia_hyper
  trigger: 22:00 + stamina >= 90 + 隨機 30%
  narrative: "你躺下，但心臟像戰鼓。今天練得太狠了。"
  choices:
    [ 硬睡 ]           → 明早 stamina 只恢復 60%, mood -5
    [ 起身夜訓 ]        → 多一次訓練機會, 50% 被發現
    [ 看星空 ]         → stamina 不恢復, mood +20, WIL +0.3
```

### 事件範例：擺爛

```js
event: training_slacker
  trigger: 訓練動作 + stamina <= 25 OR mood <= 20 + 隨機 25%
  narrative: "你舉起木劍，又放下。一切都沒意義。"

    [ 撐著繼續 ]   → 訓練效果 -30%, WIL +0.1
    [ 隨便比劃 ]   → 訓練效果 0, mood +5, 30% 被監督官發現
    [ 直接走人 ]   → 訓練效果 0, mood +10, 100% 被發現, 監督官 -10
```

## E.5 勞動請求機制 + NPC 注意系統

### 兩階段：後台累積 → 事件觸發

### 階段 1：NPC 默默注意（後台 flag 累積）

每次玩家執行動作時，系統掃描在場 NPC 是否「注意到」某些行為：

```js
// 背後執行的累積邏輯（無 UI）
onPlayerAction(action, currentNPCs) {
  if (action.type === 'training' && currentNPCs.includes('blacksmithGra')) {
    if (action.attr === 'STR' && Stats.eff('STR') >= 15) {
      Flags.increment('gra_noticed_strength', 1);
    }
  }
  if (action.type === 'training' && currentNPCs.includes('melaKook')) {
    if (action.attr === 'AGI' && Stats.eff('AGI') >= 15) {
      Flags.increment('mela_noticed_agility', 1);
    }
  }
  // 其他 NPC 的「看到」判定...
}
```

玩家**看不到**這些 flag 在累積，直到引爆。

### 階段 2：NPC 請求主人派你去

幾天後，累積的 flag 觸發事件：

```js
event: gra_requests_porter
  trigger: flag_gte('gra_noticed_strength', 3) + 1~3 天後隨機
  priority: 2  // 主人的派遣類
  forced: true
  narrative:
    "早上一個侍從來找你。
     「主人昨晚決定：今天你去鍛造坊幫葛拉搬貨。
      葛拉跟他提過你。」"
  choices:
    [ 乖乖去 ]          → money +8, gra +10, stamina -20, 上午 2 個訓練格消失
    [ 裝病 ]            → master -10, gra -5, 上午無事但可能被識破（30%）
    [ 偷懶應付 ]        → money +3, gra -5, stamina -10
    [ 認真完成 ]        → money +10, gra +15, stamina -25, 觸發連鎖事件
```

### 階段 3：連鎖展開（可選）

認真完成後可能觸發下一個事件：

```js
event: gra_impressed
  trigger: flag 'gra_requests_porter_completed_well'
  narrative:
    "葛拉在你離開前喊住你：
     『明天如果主人沒安排，跟他說我要你。』
      他從抽屜裡拿出一個銅片遞給你。"
  effects:
    - gra +15
    - item_add 'gra_token'
    - flag 'gra_wants_regular_porter'
    - 之後 gra 類事件觸發機率 ×2
```

### 關鍵：玩家無法主動要求

**左欄永遠不會出現「去幫鐵匠搬貨」按鈕**。這個關係只能透過：
1. NPC 注意你
2. NPC 請求主人
3. 主人決定派你
4. 你決定如何應對

```js

    - 名聲快速上升（+10/三天內）

    - 或 timeline 事件

  narrative:
    "侍從走進你的房間。
     『主人要見你。別讓他等。』"

    - 自動切換場景到 masterRoom
    - 觸發主人對話事件
```

### 對話事件的選擇

```js
event: master_audience_routine
  trigger: 在主人房 + flag 'summoned_by_master'
  narrative:
    "主人坐在椅子上，手指輕敲桌面。
     『你最近的表現讓我感興趣。』"
  choices:
    [ 謙遜：『全靠主人的訓練。』]
      → master +5, 顯得乖順

      → master 0, officer +3（聽到）
    [ 沉默不語 ]
      → master -2（覺得陰鬱）, 之後事件多一個 flag 'quiet_slave'
```

### 對話結束後

```js
onEventEnd: {
  - Flags.unset('summoned_by_master')   // 關閉主人房
  - Auto-return to training ground
  - 可能觸發後續事件（賞賜/任務/警告）
}
```

**主人房一年只會「打開」幾次。** 每次都是重要時刻。

## E.8 金錢流向（主人經濟）

### 原則：**所有金錢都從主人來，玩家只能在事件中使用**

### 收入管道（全部被動）

```
① 勞動事件完成
   - 搬貨  → 3~10 銀幣
   - 幫廚  → 3~8 銀幣 + 食物
   - 跑腿  → 5 銀幣

   - 一般勝利   → 9~24 銀幣到手（30~80 的 30%）

   - 重大勝利後
   - 重要事件結束後
   - 好感累積的階段性賞賜

④ 任務報酬（未來 F4）
   - NPC 個人任務
   - 秘密任務
```

### 支出管道（全部事件觸發）

```

```

### 金錢的意義

玩家累積金錢的目的是**為了某個特定事件**，不是隨時消費。
這讓金錢有「目標感」，而不是「零碎購物」。

## E.9 事件觸發優先度

### 四個事件槽位

```
晨間事件槽（06:00~08:00，早餐前）
  - 主人派遣
  - 特殊召見
  - 天氣/世界狀態事件
  
午間事件槽（12:00~14:00，午餐期間）
  - 用餐互動事件
  - NPC 接近事件
  - 跳餐/偷食事件
  
下午事件槽（14:00~18:00，訓練期間）
  - 隊友切磋邀請
  - 監督官視察
  - 訓練意外
  
夜間事件槽（20:00~22:00，就寢前）
  - 失眠事件
  - 夜間訪客
  - 夢境
  
深夜事件槽（22:00，就寢時）
  - 夢境事件
  - 夜襲
  - 訪客
```

### 優先度規則

```
priority 1 (highest): Timeline 強制事件（Day 5/50/75/100）
priority 2:          主人直接命令事件（派遣、召見）
priority 3:          透過主人的 NPC 請求
priority 4:          隨機 NPC 接近事件
priority 5:          數值異常觸發事件（飢餓/失眠/擺爛）
priority 6:          風味環境事件（天氣、季節）
```

高優先級的事件**佔用**該時段，低優先級被跳過。

### 每日事件密度預估

```
平均一天觸發事件數（正常遊玩）：
  晨間：  20% (派遣/召見)
  午間：  40% (用餐互動)
  下午：  30% (訓練中事件)
  夜間：  20% (就寢前)
  深夜：  15% (夢境)
  
  → 每天約 1~3 個事件
```

##### JavaScript API（main.js 內 IIFE）

```js
const ChoiceModal = (() => {
  let _activeEvent = null;

  function show(ev) {
    _activeEvent = ev;
    _render(ev);
    document.getElementById('modal-choice').classList.add('open');
    GameState.setPaused(true);      // 選擇時暫停時間推進
  }

  function _render(ev) { /* 填 title/body/buttons */ }

  function _onChoose(choice) {
    // 1. 解析 effects（直接）或 rolls（加權隨機）
    const resolved = choice.rolls ? _rollWeighted(choice.rolls) : choice;
    // 2. 套效果
    if (resolved.effects) {
      Effects.apply(resolved.effects, { source: 'choice:'+_activeEvent.id+':'+choice.id });
    }
    // 3. 輸出敘事
    const log = resolved.log || resolved.resultLog;
    if (log) Log.write(log);
    // 4. onAilment 等擴展鉤子處理
    if (resolved.onAilment) Stats.addAilment(resolved.onAilment);
    // 5. 關閉
    _close();
  }

  function _close() {
    document.getElementById('modal-choice').classList.remove('open');
    GameState.setPaused(false);
    const cb = _activeEvent?.onClose;
    _activeEvent = null;
    if (cb) cb();
  }

  function _rollWeighted(rolls) {
    const total = rolls.reduce((s, r) => s + r.weight, 0);
    let n = Math.random() * total;
    for (const r of rolls) { n -= r.weight; if (n <= 0) return r; }
    return rolls[rolls.length - 1];
  }

  return { show };
})();
```

##### 時間/流程阻塞（關鍵設計）

**問題**：用餐事件是在 `_resolveNonTrainingSlots()` 的 loop 裡強制觸發的。
跳 modal 後 loop 必須暫停，否則時段會直接推進過去。

**解法**：引入 `GameState.paused` 標記
- `ChoiceModal.show` 時設 `GameState.setPaused(true)`
- `_resolveNonTrainingSlots` 每次推進前檢查 paused，為 true 則 `return`（中斷 loop）
- modal 關閉時 paused=false，由玩家下一次動作（或 sleep 按鈕）重新進入 loop

**為什麼不用 async/Promise**：改動過大，要把整條 action pipeline 改成非同步。
pause flag 方案改動最小、最易驗證。

## 最後：這個設計的原創性

這個「奴隸循環」的設計組合了：

- **Pathologic** 的絕望時間壓力
- **Disco Elysium** 的選擇式敘事
- **Darkest Dungeon** 的狀態崩壞機制
- **Papers, Please** 的被動官僚化
- **實況野球高校篇** 的屬性訓練循環
- **Undertale** 的和平 vs 暴力路線

但組合起來形成的「**奴隸主題 Narrative Roguelite**」是獨創的。
沒有任何現有遊戲完全做到這個核心體驗。
